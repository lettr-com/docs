# Sending Domains

Sending domains are the primary domains you use to send emails. When you send an email from `hello@example.com`, `example.com` is your sending domain. Proper configuration ensures your emails reach the inbox and are trusted by recipients.

## Why Verify Sending Domains?

Email providers are increasingly strict about authentication. Verified sending domains provide:

- **Deliverability** - Authenticated emails are trusted and reach the inbox
- **Brand Identity** - Send from your own domain, not a shared one
- **Reputation Building** - Build a positive sending reputation for your domain
- **Compliance** - Meet modern email authentication requirements (SPF, DKIM, DMARC)

## Adding a Sending Domain

<Steps>
  <Step title="Add Domain in Dashboard">
    Go to **Domains → Add Domain** and enter your domain name.
  </Step>
  <Step title="Configure DNS Records">
    Add the SPF and DKIM records to your domain's DNS settings.
  </Step>
  <Step title="Verify Domain">
    Click **Verify** to confirm DNS propagation. Verification typically takes a few minutes but can take up to 48 hours.
  </Step>
</Steps>

## Via API

```javascript
const domain = await lettr.domains.create({
  name: 'example.com',
  type: 'sending'
});

// View required DNS records
for (const record of domain.dnsRecords) {
  console.log(`${record.type} | ${record.host} | ${record.value}`);
}
```

## Required DNS Records

You need to add three DNS records for email authentication:

### SPF Record

Authorizes Lettr's servers to send email on behalf of your domain:

```
Type:  TXT
Host:  @ (or leave blank for root domain)
Value: v=spf1 include:spf.lettr.dev ~all
```

<Tip>
  If you already have an SPF record, merge Lettr's include with your existing record. Learn more in our [SPF guide](/learn/domains/spf).
</Tip>

### DKIM Record

Enables cryptographic signing to verify email authenticity:

```
Type:  CNAME
Host:  lettr._domainkey
Value: dkim.lettr.dev
```

### DMARC Record

DMARC is required for sending domains. Add a DMARC TXT record:

```
Type:  TXT
Host:  _dmarc
Value: v=DMARC1; p=none; rua=mailto:dmarc@example.com
```

See our [DMARC guide](/learn/domains/dmarc) for detailed configuration and safe rollout to enforcement.

## Custom Return-Path and Bounce Record

- **Primary (root) domains** use an **SPF record** (TXT) to authorize sending and bounces.
- **Subdomains** use a **Bounce record**: a CNAME custom return-path. Add a CNAME with value `sparkpostmail.com` for the subdomain you use for the Return-Path (e.g. `bounce` or `send`):

```
Type:  CNAME
Host:  bounce (or send / your chosen subdomain)
Value: sparkpostmail.com
```

Configure via API:

```javascript
const domain = await lettr.domains.create({
  name: 'mail.example.com',
  type: 'sending',
  customReturnPath: 'bounce'
});
```

## Verify DNS Records

```javascript
const result = await lettr.domains.verify('dom_123');

if (result.verified) {
  console.log('Domain verified successfully!');
} else {
  console.log('Verification status:', result.status);
  for (const record of result.missingRecords) {
    console.log(`Missing: ${record.type} record at ${record.host}`);
  }
}
```

## Sending From Verified Domains

Once verified, you can send from any email address on your domain:

```javascript
await lettr.emails.send({
  from: 'hello@example.com',
  to: ['recipient@other.com'],
  subject: 'Hello from Lettr',
  html: '<p>Your email content here.</p>'
});

// All addresses on the domain work:
// - hello@example.com ✓
// - support@example.com ✓
// - team@example.com ✓
// - any-address@example.com ✓
```

## Sender Name

Customize the display name that appears alongside your email address:

```javascript
// String format
await lettr.emails.send({
  from: 'Acme Inc <hello@example.com>',
  to: ['recipient@other.com'],
  subject: 'Hello',
  html: '<p>Hello!</p>'
});

// Object format
await lettr.emails.send({
  from: {
    email: 'hello@example.com',
    name: 'Acme Inc'
  },
  to: ['recipient@other.com'],
  subject: 'Hello',
  html: '<p>Hello!</p>'
});
```

## Subdomains for Sending

We recommend using subdomains to isolate different types of email:

| Subdomain | Use Case | Benefit |
|-----------|----------|---------|
| `mail.example.com` | Transactional emails | Isolated reputation |
| `news.example.com` | Marketing newsletters | Separate reputation |
| `alerts.example.com` | Automated notifications | Protected main domain |

<Tip>
  If one email type experiences deliverability issues (e.g., high unsubscribes from marketing), it won't affect your transactional emails on a different subdomain.
</Tip>

### Setting Up a Subdomain

Add the subdomain in Lettr the same way as a root domain:

```javascript
const domain = await lettr.domains.create({
  name: 'mail.example.com',
  type: 'sending'
});
```

DNS records for subdomains:

```
# SPF
Type:  TXT
Host:  mail
Value: v=spf1 include:spf.lettr.dev ~all

# DKIM
Type:  CNAME
Host:  lettr._domainkey.mail
Value: dkim.lettr.dev
```

## Domain Status

Check your domain's verification status:

```javascript
const domain = await lettr.domains.get('dom_123');

console.log({
  name: domain.name,
  status: domain.status,         // 'pending', 'verified', 'failed'
  spfStatus: domain.spfStatus,   // 'pending', 'verified', 'failed'
  dkimStatus: domain.dkimStatus, // 'pending', 'verified', 'failed'
  createdAt: domain.createdAt
});
```

## Troubleshooting

| Issue | Solution |
|-------|----------|
| SPF not verifying | Ensure TXT record is on root domain (`@`) |
| DKIM not verifying | Check CNAME record at `lettr._domainkey` |
| Sending fails | Verify domain status is `verified` |
| Emails going to spam | Ensure DMARC is configured and enforced |
