# Sending Domains

Sending domains are the primary domains you use to send emails. When you send an email from `hello@example.com`, `example.com` is your sending domain. Proper configuration ensures your emails reach the inbox and are trusted by recipients.

## Why Verify Sending Domains?

Email providers are increasingly strict about authentication. Verified sending domains provide:

- **Deliverability** - Authenticated emails are trusted and reach the inbox
- **Brand Identity** - Send from your own domain, not a shared one
- **Reputation Building** - Build a positive sending reputation for your domain
- **Compliance** - Meet modern email authentication requirements (SPF, DKIM, DMARC)

## Adding a Sending Domain

<Steps>
  <Step title="Add domain in Dashboard">
    Go to **Domains → Sending Domains** and click **+ New Sending Domain**. Enter your domain name.
  </Step>
  <Step title="Configure DNS Records">
    Add the required DNS records (see table below—root domains need SPF, DKIM, DMARC; subdomains need DKIM, DMARC, and Bounce CNAME only).
  </Step>
  <Step title="Verify Domain">
    Click **Verify** to confirm DNS propagation. Verification typically takes a few minutes but can take up to 48 hours.
  </Step>
</Steps>

## Via API

Create a sending domain with `POST /domains`. The request body contains only the domain name:

```javascript
const response = await fetch('https://app.lettr.com/api/domains', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': 'Bearer YOUR_API_KEY'
  },
  body: JSON.stringify({ domain: 'example.com' })
});

const { message, data } = await response.json();
// data.domain, data.status, data.status_label, data.dkim (selector, public, headers)
```

On success (201), use `data.dkim` to add your DKIM DNS record. For full DNS record details (including CNAME for the return-path), call [Get Domain](/api-reference/domains/get-domain) (`GET /domains/{domain}`). See [Create Domain](/api-reference/domains/create-domain) for the full request/response schema.

## Required DNS Records

Required records differ for a **primary (root) domain** (e.g. `example.com`) vs a **subdomain** (e.g. `mail.example.com`). For subdomains, SPF for the return-path is handled automatically—you only add the Bounce CNAME.

<Tip>
  **We recommend using a subdomain** (e.g. `mail.example.com`, `news.example.com`) for sending. Subdomains isolate your sending reputation from your root domain and make it easier to add the Bounce CNAME required for return-path.
</Tip>

### Primary (root) domain — e.g. `example.com`

| Record | Type | Host | Value |
|--------|------|------|-------|
| **SPF** | TXT | `@` (or blank) | `v=spf1 include:_spf.sparkpostmail.com ~all` |
| **DKIM** | CNAME | `lettr._domainkey` | (from Get Domain / create response) |
| **DMARC** | TXT | `_dmarc` | `v=DMARC1;p=none;` |

If you already have an SPF record, merge the include: `v=spf1 include:_spf.sparkpostmail.com ... ~all`. Learn more in our [SPF guide](/learn/domains/spf).

### Subdomain — e.g. `mail.example.com`

| Record | Type | Host | Value |
|--------|------|------|-------|
| **DKIM** | CNAME | `lettr._domainkey.mail` | (from Get Domain / create response) |
| **DMARC** | TXT | `_dmarc` | `v=DMARC1;p=none;` |
| **Bounce** | CNAME | `bounce` (or `send`) | `sparkpostmail.com` |

No SPF record is needed for the subdomain—return-path (bounce) is handled automatically via the Bounce CNAME. See [Custom Return-Path and Bounce Record](#custom-return-path-and-bounce-record) below.

See our [DMARC guide](/learn/domains/dmarc) for detailed configuration and safe rollout to enforcement.

## Custom Return-Path and Bounce Record

- **Primary (root) domains** use an **SPF record** (TXT) to authorize sending and bounces.
- **Subdomains** use a **Bounce record** (CNAME to `sparkpostmail.com`) for the return-path; SPF for the subdomain is not required and is handled automatically. Add the Bounce CNAME for the subdomain you use (e.g. `bounce` or `send`):

```
Type:  CNAME
Host:  bounce (or send / your chosen subdomain)
Value: sparkpostmail.com
```

Create the subdomain via API the same way (body `{ "domain": "mail.example.com" }`). Add the Bounce CNAME record in your DNS provider as shown above.

## Verify DNS Records

Trigger a verification check with `POST /domains/{domain}/verify`:

```javascript
const response = await fetch('https://app.lettr.com/api/domains/example.com/verify', {
  method: 'POST',
  headers: { 'Authorization': 'Bearer YOUR_API_KEY' }
});

const { message, data } = await response.json();
// data.domain, data.dkim_status, data.cname_status, data.ownership_verified
// data.dns (dkim_record, cname_record, dkim_error, cname_error) when unverified
```

## Sending From Verified Domains

Once verified, you can send from any email address on your domain:

```javascript
await lettr.emails.send({
  from: 'hello@example.com',
  to: ['recipient@other.com'],
  subject: 'Hello from Lettr',
  html: '<p>Your email content here.</p>'
});

// All addresses on the domain work:
// - hello@example.com ✓
// - support@example.com ✓
// - team@example.com ✓
// - any-address@example.com ✓
```

## Sender Name

Customize the display name that appears alongside your email address:

```javascript
// String format
await lettr.emails.send({
  from: 'Acme Inc <hello@example.com>',
  to: ['recipient@other.com'],
  subject: 'Hello',
  html: '<p>Hello!</p>'
});

// Object format
await lettr.emails.send({
  from: {
    email: 'hello@example.com',
    name: 'Acme Inc'
  },
  to: ['recipient@other.com'],
  subject: 'Hello',
  html: '<p>Hello!</p>'
});
```

## Subdomains for Sending

We recommend using subdomains to isolate different types of email:

| Subdomain | Use Case | Benefit |
|-----------|----------|---------|
| `mail.example.com` | Transactional emails | Isolated reputation |
| `news.example.com` | Marketing newsletters | Separate reputation |
| `alerts.example.com` | Automated notifications | Protected main domain |

<Tip>
  If one email type experiences deliverability issues (e.g., high unsubscribes from marketing), it won't affect your transactional emails on a different subdomain.
</Tip>

### Setting Up a Subdomain

Add the subdomain via the same Create Domain endpoint with the subdomain name:

```javascript
const response = await fetch('https://app.lettr.com/api/domains', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': 'Bearer YOUR_API_KEY'
  },
  body: JSON.stringify({ domain: 'mail.example.com' })
});
```

DNS records for subdomains: DKIM, DMARC, and Bounce CNAME as in the [Required DNS Records](#required-dns-records) table (no SPF needed for subdomains).

## Domain Status

Retrieve a domain's status and DNS verification state with `GET /domains/{domain}`:

```javascript
const response = await fetch('https://app.lettr.com/api/domains/example.com', {
  headers: { 'Authorization': 'Bearer YOUR_API_KEY' }
});

const { data } = await response.json();
// data.domain, data.status ('pending' | 'approved' | 'blocked'), data.status_label
// data.can_send, data.dkim_status, data.cname_status, data.dns, data.created_at, data.updated_at
```

## Troubleshooting

| Issue | Solution |
|-------|----------|
| SPF not verifying | Ensure TXT record is on root domain (`@`) |
| DKIM not verifying | Check CNAME record at `lettr._domainkey` |
| Sending fails | Verify domain status is `verified` |
| Emails going to spam | Ensure DMARC is configured and enforced |
