# Introduction

Domains are the foundation of your email infrastructure in Lettr. Properly configured domains ensure your emails reach the inbox, maintain your brand identity, and comply with email authentication standards.

<Warning>
  Domain not verifying? Check our [troubleshooting guide](/knowledge-base/troubleshooting/authentication) for common DNS issues.
</Warning>

## Types of Domains

Lettr supports four types of domains, each serving a specific purpose:

<CardGroup cols={2}>
  <Card title="Sending Domains" icon="paper-plane" href="/learn/domains/sending-domains">
    Primary domains used for sending emails. Requires SPF and DKIM authentication.
  </Card>
  <Card title="Tracking Domains" icon="chart-line" href="/learn/domains/tracking-domains">
    Custom domains for click and open tracking links instead of using Lettr's default tracking domain.
  </Card>
  <Card title="Inbound Domains" icon="inbox" href="/learn/domains/inbound-domains">
    Domains configured to receive emails and process them via webhooks.
  </Card>
  <Card title="Storage Domains" icon="image" href="/learn/domains/storage-domains">
    Custom domains for hosting images embedded in your email HTML.
  </Card>
</CardGroup>

## Domain Comparison

| Domain Type | Purpose | Required DNS Records |
|-------------|---------|---------------------|
| Sending | Send emails from your domain | SPF (TXT) or CNAME, DKIM (CNAME), DMARC |
| Tracking | Custom tracking links | CNAME (SSL), CNAME (proxy) |
| Inbound | Receive emails | MX records pointing to Lettr |
| Storage | Host email images | CNAME (SSL), CNAME (proxy) |

## Email Authentication

Email authentication protocols verify that emails are legitimately sent by the domain owner. Lettr requires proper authentication for all sending domains.

<CardGroup cols={2}>
  <Card title="SPF" icon="list-check" href="/learn/domains/spf">
    Sender Policy Framework - Authorize Lettr's servers to send on your behalf.
  </Card>
  <Card title="DKIM" icon="key" href="/learn/domains/dkim">
    DomainKeys Identified Mail - Cryptographically sign emails to prove authenticity.
  </Card>
  <Card title="DMARC" icon="shield" href="/learn/domains/dmarc">
    Domain-based Message Authentication - Define policies for failed authentication.
  </Card>
  <Card title="BIMI" icon="image" href="/learn/domains/bimi">
    Brand Indicators for Message Identification - Display your logo in email clients.
  </Card>
</CardGroup>

## Quick Setup

<Steps>
  <Step title="Add domain in Dashboard">
    Go to **Domains**, open the tab for your domain type (e.g. **Sending Domains**), and click **+ New Sending Domain** (or **+ New Tracking Domain**, **+ New Inbound Domain**, **+ New Storage Domain**). Enter your domain name.
  </Step>
  <Step title="Add DNS Records">
    Add the provided DNS records to your domain's DNS settings. Records vary by domain type.
  </Step>
  <Step title="Verify Domain">
    Click **Verify** to confirm DNS propagation. Verification typically takes a few minutes but can take up to 48 hours.
  </Step>
</Steps>

## Add Domain via API

Register a new sending domain with `POST /domains`. The request body contains only the domain name:

```javascript
const response = await fetch('https://app.lettr.com/api/domains', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': 'Bearer YOUR_API_KEY'
  },
  body: JSON.stringify({ domain: 'example.com' })
});

const { message, data } = await response.json();
// data.domain, data.status, data.status_label, data.dkim (selector, public, headers)
```

On success (201), the response includes the domain and DKIM details so you can add the required DNS records. Use [Get Domain](/api-reference/domains/get-domain) to retrieve full DNS record details (including CNAME) for the domain.

## Domain Status

Domains can have different statuses during and after verification:

| Status | Description |
|--------|-------------|
| `not_started` | Domain added but verification not initiated |
| `pending` | DNS records not yet detected, verification in progress |
| `verified` | All records verified and domain is active |
| `failed` | Verification failed after 72 hours (check DNS records) |
| `temporary_failure` | Previously verified domain with DNS issues detected |

<Info>
  For `temporary_failure` status, Lettr will recheck DNS records for 72 hours. If records are detected, status returns to `verified`. Otherwise, it changes to `failed`.
</Info>

## Check Domain Status

```javascript
const domain = await lettr.domains.get('dom_123');

console.log({
  name: domain.name,
  type: domain.type,
  status: domain.status,
  spfStatus: domain.spfStatus,
  dkimStatus: domain.dkimStatus,
  dmarcStatus: domain.dmarcStatus
});
```

## Subdomains vs Root Domains

We recommend using subdomains for sending emails to isolate your sending reputation:

| Subdomain | Use Case |
|-----------|----------|
| `mail.example.com` | Transactional emails |
| `news.example.com` | Marketing newsletters |
| `updates.example.com` | Product updates |

<Tip>
  Using subdomains protects your root domain's reputation. If one subdomain experiences deliverability issues, your other subdomains remain unaffected.
</Tip>
